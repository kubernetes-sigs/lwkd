## Featured PRs

### [139218: Wrap errors from pod group preemption](https://github.com/kubernetes/kubernetes/pull/139218)

[Argh4k](https://github.com/Argh4k) has landed a reader-facing improvement to kube-scheduler error messages: every error emitted by the pod-group preemption path is now prefixed with `pod group preemption:` so operators can immediately tell which preemption flow produced a given `SchedulerError`. This mirrors the [existing behavior](https://github.com/kubernetes/kubernetes/blob/ea692abff6bb8ed2b7e20d7e27de76f77bf083d6/pkg/scheduler/framework/plugins/defaultpreemption/default_preemption.go#L158-L164) of the default per-pod preemption path, which already prefixes errors with `preemption:`, and is part of the v1.37 milestone under SIG Scheduling.

The change is part of [KEP-5710: Workload-Aware Preemption](https://github.com/kubernetes/enhancements/tree/master/keps/sig-scheduling/5710-workload-aware-preemption#delayed-preemption), which introduces a parallel preemption path for pod groups (gang-scheduled workloads) alongside the existing per-pod preemption. With two preemption flows now writing into the same `SchedulerError` stream, error logs that previously could only have come from default per-pod preemption can now come from either path, and without a prefix, it was effectively impossible to tell which one. This is especially painful when triaging gang-scheduling failures in batch and AI/ML workloads, where pod-group preemption is the common case.


[139251: Rename Signal enum keys in CRI API](https://github.com/kubernetes/kubernetes/pull/139251)

[SergeyKanzhelev](https://github.com/SergeyKanzhelev) has renamed all enum keys in the `Signal` type within `api.proto` to carry a `SIGNAL_` prefix (e.g., `SIGABRT` becomes `SIGNAL_SIGABRT`). This resolves a [long-standing naming collision](https://github.com/kubernetes/kubernetes/issues/138826) between the CRI API's enum identifiers and standard C signal macros, which caused build failures in environments where C signal headers were present.

The change is part of [KEP-4960: ContainerStopSignals](https://github.com/kubernetes/enhancements/blob/master/keps/sig-node/4960-container-stop-signals/README.md) (introduced in v1.33, currently Alpha). Because the feature hasn't yet reached Beta, the rename was done directly without a deprecation period. While this is a breaking change for developers, anyone building against cri-api will need to update their code and rebuild binaries; the underlying integer values are unchanged, so the wire format remains compatible and cross-version communication is unaffected. Downstream projects like [containerd](https://github.com/containerd/containerd) and [CRI-O](https://github.com/cri-o/cri-o) will need to update their internal references to match the new names.

## KEP of the Week

[KEP-2535: Ensure Secret Pulled Images](https://github.com/kubernetes/enhancements/tree/master/keps/sig-node/2535-ensure-secret-pulled-images)

This enhancement improves Kubernetes image security by ensuring that access to container images is authenticated even when images are already present on a node. It introduces kubelet-based image verification for `IfNotPresent` and `Never` pull policies, allowing administrators to enforce authorization checks without requiring the `Always` pull policy. The feature provides configurable verification modes, strengthens control over private image access, and addresses long-standing security concerns around image reuse while reducing dependence on registry availability during container startup.

KEP-2535 graduated to Beta in Kubernetes v1.35, strengthening image access security through credential-aware verification of previously pulled images and enhanced kubelet credential caching.

## Other Merges

* The [MaxUnavailableStatefulSet feature](https://github.com/kubernetes/kubernetes/pull/139466) is now enabled by default.
* kubeadm: [fixed kubeadm init phase certs --dry-run](https://github.com/kubernetes/kubernetes/pull/139339) to correctly copy existing CA files.
* Errors coming from pod group preemption are now prefixed with [`pod group preemption:` message](https://github.com/kubernetes/kubernetes/pull/139218).
* Enum keys of [Signal are now prefixed with `SIGNAL_`](https://github.com/kubernetes/kubernetes/pull/139251) in api.proto definition to avoid conflicts with C++ macroses.
* Fixed a regression in 1.36 where modifications to scheduling directives (nodeSelector, tolerations, node affinity) on suspended [Jobs were rejected if the JobSuspended condition](https://github.com/kubernetes/kubernetes/pull/139287) had not yet been set by the job controller.
* Fixed a [race condition in preemption](https://github.com/kubernetes/kubernetes/pull/139162), where a preemptor pod could get stuck in unschedulable state.
* Fixed a bug where [Pods that share multi-node claims and also have per-node claims](https://github.com/kubernetes/kubernetes/pull/139017) can get stuck in Pending.
* Kubelet now [recovers from corrupted subpath mount points](https://github.com/kubernetes/kubernetes/pull/138856) (e.g. stale NFS file handle) during container restart instead of leaving the pod stuck in CreateContainerConfigError. 
* kube-scheduler: [Added `PlacementCycleState` to the scheduling framework](https://github.com/kubernetes/kubernetes/pull/138274), providing per-placement state to `PlacementScore` plugins under the alpha `TopologyAwareWorkloadScheduling` feature gate.
