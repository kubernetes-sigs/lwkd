## Featured PRs

### [139218: Wrap errors from pod group preemption](https://github.com/kubernetes/kubernetes/pull/139218)

[Argh4k](https://github.com/Argh4k) has landed a reader-facing improvement to kube-scheduler error messages: every error emitted by the pod-group preemption path is now prefixed with `pod group preemption:` so operators can immediately tell which preemption flow produced a given `SchedulerError`. This is scoped to SIG Scheduling, and is targeted at the v1.37 milestone.

The change is part of [KEP-5710: Workload-Aware Preemption](https://github.com/kubernetes/enhancements/tree/master/keps/sig-scheduling/5710-workload-aware-preemption#delayed-preemption), which introduces a parallel preemption path for pod groups (gang-scheduled workloads) alongside the existing per-pod preemption. With two preemption flows now writing into the same `SchedulerError` stream, error logs that previously could only have come from default per-pod preemption can now come from either path, and without a prefix, it was effectively impossible to tell which one. This is especially painful when triaging gang-scheduling failures in batch and AI/ML workloads, where pod-group preemption is the common case.

### [139251: Rename Signal enum keys in CRI API](https://github.com/kubernetes/kubernetes/pull/139251)
[SergeyKanzhelev](https://github.com/SergeyKanzhelev) has implemented a source-level breaking change to the CRI API to resolve long-standing naming collisions (https://github.com/kubernetes/kubernetes/issues/138826) with C++ macros. All enum keys in the
Signal type within api.proto now carry a SIGNAL_ prefix (e.g., SIGABRT becomes SIGNAL_SIGABRT). This ensures that the CRI API can be reliably compiled in environments where standard C signal headers are present, which previously caused build failures due to macro expansion.

This change is part of the ContainerStopSignals feature (introduced in v1.33). While this is a breaking change for developers—requiring anyone building against the cri-api to update their code and rebuild binaries—the underlying integer values remain unchanged. As a result, the wire format remains compatible, and there is no impact on cross-version communication. Because the feature is currently in Alpha, the rename was performed directly without a deprecation period to ensure the API is clean before advancing to Beta. Downstream projects like containerd and CRI-O will need to update their internal references to match these new names.
